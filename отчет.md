## <a name="11"></a>1. Практична робота №1.1

### 1.1 Опис предметної області

**Служба доставки малогабаритних вантажів** — це система для обліку та відстеження посилок невеликого розміру та ваги. Система охоплює такі бізнес-процеси:

- реєстрація клієнтів (відправників/отримувачів);
- приймання та реєстрація посилок із присвоєнням унікального трек-номера;
- призначення кур'єра на доставку;
- відстеження статусів доставки (прийнято, в обробці, передано в доставку, доставлено);
- фіксація платежів за доставку.

### 1.2 Перелік сутностей

| № | Сутність | Призначення |
|---|----------|-------------|
| 1 | `Employees` | Працівники служби доставки (кур'єри, оператори, менеджери) |
| 2 | `Customers` | Клієнти, які замовляють доставку |
| 3 | `Parcels` | Посилки з трек-номером, вагою, статусом |
| 4 | `Deliveries` | Доставки з призначенням кур'єра та датами |
| 5 | `PaymentLog` | Журнал платежів за виконані доставки |

## <a name="21"></a>2. Практична робота №2.1

### 2.1 Опис логіки запитів

Розроблено 4 збережені процедури для читання даних, які вирішують наступні бізнес-задачі:

1. **GetAllCustomers** — отримання повного списку клієнтів (довідка).
2. **SearchParcelByTracking** — пошук посилки за частиною трек-номера (для оператора або клієнта).
3. **GetCustomerDeliveries** — перегляд історії доставок конкретного клієнта за період.
4. **GetDeliveryReport** — аналітичний звіт по кур'єрах: кількість доставок, сумарна виручка, середня вартість.

### 2.2 SQL-скрипти процедур

```sql
-- Рівень 1: Повний список клієнтів
CREATE PROCEDURE CustomerService.GetAllCustomers
AS
BEGIN
    SELECT CustomerID, FullName, Phone, Email, Address
    FROM CustomerService.Customers
    ORDER BY FullName;
END;
GO

-- Рівень 2: Пошук посилки за трек-номером (частковий збіг)
CREATE PROCEDURE Logistics.SearchParcelByTracking
    @TrackingNumber NVARCHAR(30)  -- вхідний параметр: частина трек-номера
AS
BEGIN
    SELECT p.ParcelID, p.TrackingNumber, p.WeightKg, p.Status,
           c.FullName AS CustomerName, p.RecipientAddress
    FROM CustomerService.Parcels p
    JOIN CustomerService.Customers c ON p.CustomerID = c.CustomerID
    WHERE p.TrackingNumber LIKE '%' + @TrackingNumber + '%';
END;
GO

-- Рівень 2: Доставки клієнта за період (з опціональними датами)
CREATE PROCEDURE CustomerService.GetCustomerDeliveries
    @CustomerID INT,              -- ID клієнта
    @StartDate DATE = NULL,       -- початок періоду (необов'язково)
    @EndDate DATE = NULL          -- кінець періоду (необов'язково)
AS
BEGIN
    SELECT d.DeliveryID, p.TrackingNumber, d.Status, d.Cost,
           d.PickupDate, d.DeliveryDate
    FROM Logistics.Deliveries d
    JOIN CustomerService.Parcels p ON d.ParcelID = p.ParcelID
    WHERE p.CustomerID = @CustomerID
      AND (@StartDate IS NULL OR d.PickupDate >= @StartDate)
      AND (@EndDate IS NULL OR d.PickupDate <= @EndDate)
    ORDER BY d.PickupDate DESC;
END;
GO

-- Рівень 3: Звіт по кур'єрах з агрегацією (JOIN 4 таблиць)
CREATE PROCEDURE Logistics.GetDeliveryReport
    @StartDate DATE = NULL,   -- початок періоду
    @EndDate DATE = NULL      -- кінець періоду
AS
BEGIN
    SELECT
        e.FullName AS CourierName,
        COUNT(d.DeliveryID) AS TotalDeliveries,
        SUM(d.Cost) AS TotalRevenue,
        AVG(d.Cost) AS AvgCost
    FROM Logistics.Deliveries d
    JOIN CustomerService.Parcels p ON d.ParcelID = p.ParcelID
    JOIN HumanResources.Employees e ON d.EmployeeID = e.EmployeeID
    LEFT JOIN Finance.PaymentLog pl ON d.DeliveryID = pl.DeliveryID
    WHERE (@StartDate IS NULL OR d.PickupDate >= @StartDate)
      AND (@EndDate IS NULL OR d.PickupDate <= @EndDate)
    GROUP BY e.FullName
    HAVING COUNT(d.DeliveryID) > 0
    ORDER BY TotalRevenue DESC;
END;
GO
```

### 2.3 Скрипти виклику процедур (EXEC)

```sql

-- 2. Пошук посилки за трек-номером

-- 3. Доставки клієнта з ID=1 за весь період
EXEC CustomerService.GetCustomerDeliveries @CustomerID = 1;

-- 4. Доставки клієнта з ID=1 за січень-червень 2025

-- 5. Аналітичний звіт

-- 6. Звіт за конкретний період
```

## <a name="31"></a>3. Практична робота №3.1

### 3.1 Постановка задачі

Для служби доставки необхідно реалізувати механізми модифікації даних:

- **INSERT**: додавання нової посилки (з перевіркою на дублікат трек-номера).
- **UPDATE**: оновлення статусу доставки (з автоматичним проставлянням дати доставки).
- **DELETE**: безпечне видалення посилки (з перевіркою наявності пов'язаних доставок).
- **Scalar function**: розрахунок вартості доставки залежно від ваги, відстані та терміновості.
- **Transaction**: комплексне оформлення доставки з одночасною зміною трьох таблиць.

### 3.2 Лістинг SQL-коду

```sql
-- ===== Рівень 1: INSERT =====
CREATE PROCEDURE Logistics.AddParcel
    @TrackingNumber NVARCHAR(30),
    @WeightKg DECIMAL(8,2),
    @Description NVARCHAR(500) = NULL,
    @CustomerID INT,
    @RecipientAddress NVARCHAR(300)
AS
BEGIN
    IF EXISTS (SELECT 1 FROM CustomerService.Parcels WHERE TrackingNumber = @TrackingNumber)
    BEGIN
        THROW 50001, N'Посилка з таким трек-номером вже існує.', 1;
        RETURN;
    END

    INSERT INTO CustomerService.Parcels (TrackingNumber, WeightKg, Description, CustomerID, RecipientAddress)
    VALUES (@TrackingNumber, @WeightKg, @Description, @CustomerID, @RecipientAddress);

    PRINT N'Посилку додано. ID: ' + CAST(SCOPE_IDENTITY() AS NVARCHAR);
END;
GO

-- ===== Рівень 1: UPDATE =====
CREATE PROCEDURE Logistics.UpdateDeliveryStatus
    @DeliveryID INT,
    @NewStatus NVARCHAR(50)
AS
BEGIN
    IF NOT EXISTS (SELECT 1 FROM Logistics.Deliveries WHERE DeliveryID = @DeliveryID)
    BEGIN
        THROW 50002, N'Доставку з таким ID не знайдено.', 1;
        RETURN;
    END

    UPDATE Logistics.Deliveries
    SET Status = @NewStatus,
        DeliveryDate = CASE WHEN @NewStatus = N'Доставлено' THEN GETDATE() ELSE DeliveryDate END
    WHERE DeliveryID = @DeliveryID;
END;
GO

-- ===== Рівень 1: DELETE =====
CREATE PROCEDURE Logistics.DeleteParcel
    @ParcelID INT
AS
BEGIN
    IF NOT EXISTS (SELECT 1 FROM CustomerService.Parcels WHERE ParcelID = @ParcelID)
    BEGIN
        PRINT N'Посилку з таким ID не знайдено.';
        RETURN;
    END

    IF EXISTS (SELECT 1 FROM Logistics.Deliveries WHERE ParcelID = @ParcelID)
    BEGIN
        THROW 50003, N'Неможливо видалити: існують пов''язані доставки.', 1;
        RETURN;
    END

    DELETE FROM CustomerService.Parcels WHERE ParcelID = @ParcelID;
    PRINT N'Посилку видалено.';
END;
GO

-- ===== Рівень 2: Скалярна функція =====
CREATE FUNCTION Logistics.CalculateDeliveryCost
(
    @WeightKg DECIMAL(8,2),
    @DistanceKm DECIMAL(10,2) = 10,
    @IsExpress BIT = 0
)
RETURNS DECIMAL(10,2)
AS
BEGIN
    DECLARE @BaseRate DECIMAL(10,2) = 50.00;
    DECLARE @Total DECIMAL(10,2);

    SET @Total = @BaseRate + (@WeightKg * 15.00) + (@DistanceKm * 5.00);

    IF @IsExpress = 1
        SET @Total = @Total * 1.5;

    RETURN @Total;
END;
GO

-- ===== Рівень 3: Комплексна транзакція =====
CREATE PROCEDURE Logistics.ProcessDelivery
    @ParcelID INT,
    @EmployeeID INT,
    @DistanceKm DECIMAL(10,2) = 10,
    @IsExpress BIT = 0
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRY
        BEGIN TRANSACTION;

        IF NOT EXISTS (SELECT 1 FROM CustomerService.Parcels WHERE ParcelID = @ParcelID)
            THROW 50010, N'Посилку не знайдено.', 1;

        IF NOT EXISTS (SELECT 1 FROM HumanResources.Employees WHERE EmployeeID = @EmployeeID)
            THROW 50011, N'Працівника не знайдено.', 1;

        DECLARE @WeightKg DECIMAL(8,2);
        DECLARE @Cost DECIMAL(10,2);

        SELECT @WeightKg = WeightKg FROM CustomerService.Parcels WHERE ParcelID = @ParcelID;
        SET @Cost = Logistics.CalculateDeliveryCost(@WeightKg, @DistanceKm, @IsExpress);

        INSERT INTO Logistics.Deliveries (ParcelID, EmployeeID, Status, Cost)
        VALUES (@ParcelID, @EmployeeID, N'В обробці', @Cost);

        DECLARE @DeliveryID INT = SCOPE_IDENTITY();

        UPDATE CustomerService.Parcels
        SET Status = N'Передано в доставку'
        WHERE ParcelID = @ParcelID;

        INSERT INTO Finance.PaymentLog (DeliveryID, Amount, PaymentMethod)
        VALUES (@DeliveryID, @Cost, N'Готівка');

        COMMIT TRANSACTION;
        PRINT N'Доставку успішно оформлено. DeliveryID = ' + CAST(@DeliveryID AS NVARCHAR);
    END TRY
    BEGIN CATCH
        IF @@TRANCOUNT > 0
            ROLLBACK TRANSACTION;

        DECLARE @ErrMsg NVARCHAR(4000) = ERROR_MESSAGE();
        RAISERROR(@ErrMsg, 16, 1);
    END CATCH
END;
GO
```

### 3.5 Опис роботи транзакційного блоку

Процедура `ProcessDelivery` виконує комплексну операцію, яка модифікує три таблиці:

1. **Logistics.Deliveries** — створюється новий запис доставки з розрахованою вартістю.
2. **CustomerService.Parcels** — статус посилки змінюється на "Передано в доставку".
3. **Finance.PaymentLog** — фіксується платіж на суму доставки.

Увесь ланцюжок операцій огорнуто в `BEGIN TRANSACTION...COMMIT`. Якщо на будь-якому етапі виникає помилка, блок `CATCH` виконує `ROLLBACK TRANSACTION`, що гарантує атомарність — жодна з таблиць не залишиться в проміжному стані.

```
BEGIN TRY
    BEGIN TRANSACTION
        → INSERT INTO Logistics.Deliveries
        → UPDATE CustomerService.Parcels
        → INSERT INTO Finance.PaymentLog
    COMMIT TRANSACTION
END TRY
BEGIN CATCH
    ROLLBACK TRANSACTION   -- відкат усіх трьох змін
    RAISERROR(...)
END CATCH
```

### 3.6 Висновки до ПР №3.1

Розроблено повний набір процедур модифікації даних для служби доставки. Процедури рівня 1 забезпечують базові операції INSERT, UPDATE, DELETE. Рівень 2 додає скалярну функцію для автоматизованого розрахунку вартості доставки та валідацію вхідних даних. Рівень 3 реалізує транзакційну процедуру `ProcessDelivery`, яка атомарно модифікує три пов'язані таблиці з використанням `TRY...CATCH...ROLLBACK`, що гарантує цілісність даних при будь-яких збоях.

## <a name="41"></a>4. Практична робота №4.1

### 4.1 Постановка задачі

Дослідити роботу транзакцій у MS SQL Server: реалізувати базову транзакцію з COMMIT, механізм відкату змін через TRY...CATCH...ROLLBACK, а також експериментально перевірити вплив рівнів ізоляції (READ UNCOMMITTED, READ COMMITTED, SERIALIZABLE) на паралельний доступ до даних.

### 4.2 SQL-код створення таблиць

```sql
-- Таблиця журналу операцій
CREATE TABLE Logistics.OperationLog
(
    LogID INT PRIMARY KEY IDENTITY(1,1),
    OperationDate DATETIME NOT NULL DEFAULT GETDATE(),
    OperationType NVARCHAR(50) NOT NULL,
    TableName NVARCHAR(100) NOT NULL,
    RecordID INT,
    Description NVARCHAR(500)
);
GO
```

### 4.3 Рівень 1 — Базова транзакція з COMMIT

```sql
BEGIN TRANSACTION;

UPDATE Logistics.Deliveries
SET Status = N'В дорозі'
WHERE DeliveryID = 1;

INSERT INTO Logistics.OperationLog (OperationType, TableName, RecordID, Description)
VALUES (N'UPDATE', N'Deliveries', 1, N'Статус доставки змінено на "В дорозі"');

COMMIT TRANSACTION;
GO
```

**Результат:** статус доставки змінено, запис у журналі створено.

### 4.4 Рівень 2 — TRY...CATCH + ROLLBACK

```sql
BEGIN TRY
    BEGIN TRANSACTION;

    DECLARE @ParcelID INT = 1;
    DECLARE @CurrentWeight DECIMAL(8,2);
    SELECT @CurrentWeight = WeightKg FROM CustomerService.Parcels WHERE ParcelID = @ParcelID;

    IF @CurrentWeight < 100
        THROW 50020, N'Недостатньо ваги посилки для операції.', 1;

    UPDATE CustomerService.Parcels
    SET WeightKg = WeightKg - 100
    WHERE ParcelID = @ParcelID;

    INSERT INTO Logistics.OperationLog (...)
    VALUES (N'UPDATE', N'Parcels', @ParcelID, N'Списано 100 кг ваги');

    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    ROLLBACK TRANSACTION;
    PRINT ERROR_MESSAGE();
END CATCH;
```

**Результат:** транзакцію скасовано, вага посилки не змінилась, запис у журнал не додано.

### 4.5 Рівень 3 — Рівні ізоляції

Експерименти виконуються у двох окремих вікнах SSMS одночасно:

| Експеримент | Рівень | Поведінка |
|---|---|---|
| **Брудне читання** | READ UNCOMMITTED | Вікно 2 бачить незафіксовані зміни Вікна 1 (WeightKg = 999.99) |
| **Блокування** | READ COMMITTED | Вікно 2 чекає, доки Вікно 1 не виконає COMMIT/ROLLBACK |
| **Range Lock** | SERIALIZABLE | Вікно 2 не може вставити нові рядки, що потрапляють у діапазон Вікна 1 |

![Isolation Levels](screenshots/isolation.png)

### 4.6 Висновки до ПР №4.1

Транзакції забезпечують атомарність операцій. COMMIT фіксує зміни, ROLLBACK повертає стан до початку. Рівні ізоляції регулюють видимість даних між паралельними транзакціями: READ UNCOMMITTED найшвидший, але допускає брудне читання; READ COMMITTED запобігає брудному читанню; SERIALIZABLE повністю ізолює транзакції ціною продуктивності.

## <a name="51"></a>5. Практична робота №5.1

### 5.1 Постановка задачі

Розробити тригери для автоматизації бізнес-правил та аудиту даних у службі доставки:
- AFTER UPDATE — журналювання змін статусів та цін
- AFTER INSERT — автооновлення статусу посилки при створенні доставки
- INSTEAD OF DELETE — логічне видалення клієнтів
- Заборона змін у неробочий час

### 5.2 SQL-скрипти тригерів

**Рівень 1 — AuditLog + AFTER UPDATE:**

```sql
CREATE TABLE Logistics.AuditLog
(
    AuditID INT PRIMARY KEY IDENTITY(1,1),
    TableName NVARCHAR(100) NOT NULL,
    Operation NVARCHAR(20) NOT NULL,
    RecordID INT NOT NULL,
    OldValue NVARCHAR(500),
    NewValue NVARCHAR(500),
    ChangeDate DATETIME NOT NULL DEFAULT GETDATE(),
    UserName NVARCHAR(128) NOT NULL DEFAULT SUSER_NAME()
);
GO

CREATE TRIGGER Logistics.TR_Deliveries_Audit
ON Logistics.Deliveries
AFTER UPDATE
AS
BEGIN
    INSERT INTO Logistics.AuditLog (TableName, Operation, RecordID, OldValue, NewValue)
    SELECT N'Deliveries', N'UPDATE', d.DeliveryID,
        N'Status: ' + ISNULL(d.Status, N'') + N', Cost: ' + ISNULL(CAST(d.Cost AS NVARCHAR), N''),
        N'Status: ' + ISNULL(i.Status, N'') + N', Cost: ' + ISNULL(CAST(i.Cost AS NVARCHAR), N'')
    FROM deleted d JOIN inserted i ON d.DeliveryID = i.DeliveryID
    WHERE d.Status <> i.Status OR d.Cost <> i.Cost;
END;
```

**Рівень 2 — AFTER INSERT + бізнес-правило:**

```sql
-- Автооновлення статусу посилки при створенні доставки
CREATE TRIGGER Logistics.TR_Deliveries_AfterInsert
ON Logistics.Deliveries
AFTER INSERT
AS
BEGIN
    UPDATE CustomerService.Parcels
    SET Status = N'Передано в доставку'
    WHERE ParcelID IN (SELECT ParcelID FROM inserted);
END;

-- Заборона зміни доставленої доставки
CREATE TRIGGER Logistics.TR_Deliveries_PreventReopen
ON Logistics.Deliveries
AFTER UPDATE
AS
BEGIN
    IF EXISTS (
        SELECT 1 FROM deleted d
        WHERE d.Status = N'Доставлено'
          AND EXISTS (SELECT 1 FROM inserted i WHERE i.DeliveryID = d.DeliveryID AND i.Status <> d.Status)
    )
    BEGIN
        ROLLBACK TRANSACTION;
        THROW 50030, N'Неможливо змінити доставку, яка вже має статус "Доставлено".', 1;
    END
END;
```

**Рівень 3 — INSTEAD OF DELETE + часова заборона:**

```sql
-- Логічне видалення клієнтів
CREATE TRIGGER CustomerService.TR_Customers_SoftDelete
ON CustomerService.Customers
INSTEAD OF DELETE
AS
BEGIN
    UPDATE CustomerService.Customers
    SET IsActive = 0
    WHERE CustomerID IN (SELECT CustomerID FROM deleted);
    PRINT N'Клієнта деактивовано (логічне видалення).';
END;

-- Заборона змін у неробочий час (20:00–08:00)
CREATE TRIGGER Logistics.TR_Deliveries_WorkingHours
ON Logistics.Deliveries
AFTER INSERT, UPDATE, DELETE
AS
BEGIN
    DECLARE @Hour INT = DATEPART(HOUR, GETDATE());
    IF @Hour >= 20 OR @Hour < 8
    BEGIN
        ROLLBACK TRANSACTION;
        THROW 50040, N'Операції з доставками дозволені лише з 08:00 до 20:00.', 1;
    END
END;
```

![AuditLog](screenshots/auditlog.png)

**Тест 1 — Оновлення статусу:** виконати `UPDATE Logistics.Deliveries SET Status = N'В дорозі' WHERE DeliveryID = 1` — перевірити AuditLog.

**Тест 2 — Помилка при зміні доставленого:** виконати `UPDATE Logistics.Deliveries SET Status = N'В обробці' WHERE DeliveryID = 1` після встановлення статусу "Доставлено" — отримати THROW 50030.

### 5.4 Висновки до ПР №5.1

Тригери дозволяють автоматизувати бізнес-правила без зміни коду застосунку. AFTER тригери виконуються після операції, INSTEAD OF — замість неї. Недолік: надмірне використання тригерів погіршує продуктивність (додаткові операції при кожному DML) та ускладнює налагодження, оскільки логіка стає "невидимою" для розробника.


## <a name="questions"></a>9. Відповіді на запитання для самоконтролю

### До ПР №1.1

**1. Чим відрізняється логічне поняття «схема» від фізичного поняття «файл бази даних»?**

Схема — це логічний контейнер для групування об'єктів БД (таблиць, процедур тощо) за функціональною ознакою або правами доступу. Файл бази даних — це фізичний файл на диску (*.mdf, *.ldf), де зберігаються дані. Одна БД може мати багато схем, але фізично всі дані зберігаються в тих самих файлах. Схеми не впливають на фізичне розташування даних.

**2. Які переваги надає використання SQL Server Management Studio у порівнянні з командним рядком?**

SSMS надає графічний інтерфейс для візуалізації структури БД (діаграми), інтелектуальне автозаповнення (IntelliSense), профілювальник запитів, візуальний перегляд планів виконання та інтерактивне створення об'єктів через майстри (Wizards).

### До ПР №2.1

**1. У чому полягає перевага збереженої процедури перед звичайним SQL-запитом, що надсилається з коду програми?**

Збережена процедура компілюється та кешує план виконання на сервері, тому наступні виклики виконуються швидше. Вона зменшує мережевий трафік (передається тільки ім'я та параметри), підвищує безпеку (доступ через EXECUTE, а не прямі привілеї на таблиці) та дозволяє інкапсулювати складну бізнес-логіку.

**2. Як передати значення за замовчуванням у параметр процедури?**

Потрібно вказати значення за замовчуванням після типу параметра: `@Param INT = NULL` або `@Param INT = 0`. При виклику такий параметр можна опустити: `EXEC Procedure @Required = 1;` — параметр `@Param` отримає значення за замовчуванням.

### До ПР №3.1

**1. У чому полягає різниця між використанням RAISERROR та THROW у процедурах?**

`THROW` — сучасний спосіб (SQL Server 2012+), простіший синтаксис, автоматично передає номер помилки та рядок. `RAISERROR` — застарілий, але дозволяє задавати рівень серйозності (severity) та стан (state). `THROW` перериває виконання безумовно, тоді як `RAISERROR` можна використовувати з різними рівнями серйозності.

**2. Як отримати значення ID щойно доданого запису всередині процедури?**

За допомогою функції `SCOPE_IDENTITY()`, яка повертає останнє значення IDENTITY, згенероване в поточному скоупі (процедурі): `DECLARE @NewID INT = SCOPE_IDENTITY();`

**3. Чому важливо використовувати транзакції при модифікації декількох таблиць одночасно?**

Транзакції забезпечують атомарність — усі зміни або виконуються повністю, або не виконуються жодна. Це запобігає ситуаціям, коли, наприклад, гроші списано з рахунку, але доставку не зареєстровано через помилку. У разі збою `ROLLBACK` повертає БД у вихідний стан.

**4. Чи можна викликати іншу збережену процедуру всередині поточної?**

Так, у SQL Server процедури можна вкладати: `EXEC AnotherProcedure @Param;`. Глибина вкладеності обмежена (до 32 рівнів).

### До ПР №4.1

**1. Що таке транзакція?**

Транзакція — це логічно завершений набір SQL-операцій, який виконується повністю або не виконується зовсім (атомарність).

**2. Для чого використовується COMMIT?**

COMMIT фіксує всі зміни, зроблені в межах поточної транзакції, роблячи їх постійними та видимими для інших транзакцій.

**3. Яке призначення ROLLBACK?**

ROLLBACK скасовує всі зміни, зроблені в межах поточної транзакції, повертаючи базу даних до стану на момент BEGIN TRANSACTION.

**4. У чому полягає принцип атомарності?**

Атомарність гарантує, що транзакція або виконується повністю (COMMIT), або не виконується взагалі (ROLLBACK) — проміжних станів не існує.

**5. Для чого використовується TRY...CATCH?**

TRY...CATCH перехоплює помилки виконання SQL. У блоці CATCH можна виконати ROLLBACK, залогуючи помилку, запобігаючи пошкодженню даних.

**6. Що таке рівень ізоляції транзакції?**

Рівень ізоляції визначає, якою мірою одна транзакція "бачить" незафіксовані зміни іншої, та які блокування накладаються на дані.

**7. Що таке брудне читання (Dirty Read)?**

Брудне читання — ситуація, коли транзакція читає дані, змінені іншою транзакцією, яка ще не виконала COMMIT. Якщо та транзакція зробить ROLLBACK, прочитані дані виявляться некоректними.

**8. Чим READ COMMITTED відрізняється від READ UNCOMMITTED?**

READ UNCOMMITTED дозволяє Dirty Read (читання незафіксованих даних) і не накладає блокувань. READ COMMITTED чекає на COMMIT/ROLLBACK іншої транзакції, гарантуючи читання лише зафіксованих даних.

**9. Чому важливо використовувати транзакції?**

Транзакції забезпечують цілісність даних при паралельному доступі, запобігають частковим оновленням при збоях та гарантують узгодженість.

**10. Які проблеми виникають при паралельному доступі до БД?**

Брудне читання (Dirty Read), неповторюване читання (Non-Repeatable Read), фантомне читання (Phantom Read), втрата оновлення (Lost Update).

### До ПР №5.1

**1. У чому принципова різниця між AFTER та INSTEAD OF тригерами?**

AFTER виконується після операції INSERT/UPDATE/DELETE. INSTEAD OF виконується *замість* операції — оригінальна дія не відбувається, якщо її не прописати в тілі тригера.

**2. Чи можна в одному тригері обробити відразу три події (INSERT, UPDATE, DELETE)?**

Так: `AFTER INSERT, UPDATE, DELETE AS BEGIN ... END`. Всередині можна перевірити, яка саме подія сталася, через `IF EXISTS (SELECT * FROM inserted) AND EXISTS (SELECT * FROM deleted)` тощо.

**3. Що станеться, якщо тригер спробує змінити таблицю, на яку він сам встановлений (рекурсія)?**

Виникає рекурсивний виклик тригера. За замовчуванням SQL Server дозволяє до 32 рівнів вкладеності, після чого викликає помилку. Рівень рекурсії можна обмежити через `RECURSIVE_TRIGGERS`.

**4. Чому тригери вважаються «невидимою» логікою і як це впливає на налагодження (debugging) застосунку?**

Тригери виконуються автоматично на сервері, часто непомітно для розробника. При налагодженні важко визначити, що саме спричинило зміну даних, оскільки тригер не видно в коді застосунку, що ускладнює пошук помилок.

### До ПР №6.1

**1. Яка головна ознака того, що клас порушує принцип SRP?**

Клас має більше ніж одну причину для зміни. Наприклад, клас змінюється через: а) нову бізнес-логіку, б) зміну формату логування, в) зміну сховища даних.

**2. Чим відрізняється "закритість для змін" від "неможливості редагування коду"?**

"Закритість для змін" (OCP) означає, що існуючий код не треба модифікувати при додаванні нової функціональності — достатньо створити новий клас, що реалізує інтерфейс. Це не заборона редагування, а архітектурний принцип.

**3. Чому принцип DIP часто називають фундаментом для юніт-тестування?**

DIP дозволяє підставляти mock-об'єкти замість реальних реалізацій через інтерфейси. Наприклад, замість справжньої БД (InMemoryParcelRepository) можна передати MockParcelRepository, що повертає тестові дані.

**4. Що таке "товстий інтерфейс" і як він шкодить системі?**

"Товстий інтерфейс" має багато методів, частина з яких не потрібна конкретним реалізаціям. Це змушує класи реалізовувати методи, які вони не використовують (порушення ISP), що призводить до зайвих залежностей і знижує модульність.

### До ПР №8.1

**1. Що таке Git та для чого він використовується?**  
Git — це розподілена система контролю версій. Використовується для відстеження змін у файлах, збереження історії проєкту, повернення до попередніх версій та організації колективної роботи.

**2. Для чого застосовується команда git init?**  
`git init` створює новий локальний Git-репозиторій у поточній директорії, створюючи приховану папку `.git` з усією службовою інформацією.

**3. Що таке коміт?**  
Коміт (commit) — це зафіксований стан проєкту в певний момент часу. Кожен коміт має унікальний хеш, автора, дату та повідомлення.

**4. Чим відрізняються git add та git commit?**  
`git add` додає зміни до індексу (staging area), готуючи їх до фіксації. `git commit` безпосередньо фіксує підготовлені зміни в репозиторії, створюючи новий коміт.

**5. Для чого використовується git status?**  
`git status` показує поточний стан робочої директорії: які файли змінено, які додано до індексу, а які не відстежуються Git.

**6. Як переглянути історію змін?**  
За допомогою `git log` (повна історія) або `git log --oneline` (скорочений формат — один рядок на коміт).

**7. Що таке віддалений репозиторій?**  
Віддалений репозиторій (remote) — це копія репозиторію, що зберігається на зовнішньому сервері (наприклад, GitHub, GitLab). Дозволяє синхронізувати роботу між розробниками.

**8. Для чого використовується git push?**  
`git push` надсилає локальні коміти у віддалений репозиторій, оновлюючи його стан.

**9. Що виконує команда git remote add origin?**  
`git remote add origin <URL>` додає посилання на віддалений репозиторій із псевдонімом `origin`, дозволяючи надсилати та отримувати зміни.

**10. Які переваги має система контролю версій?**  
Збереження історії, можливість повернення до попередніх версій, паралельна робота через гілки, колаборація через віддалені репозиторії, резервне копіювання коду.

### До ПР №9.1

**1. Що таке гілка у Git?**  
Гілка (branch) — це окрема лінія розробки. Вона дозволяє працювати над новою функцією або виправленням незалежно від основної версії проєкту.

**2. Для чого використовують гілки під час розробки?**  
Гілки дозволяють ізолювати різні напрямки роботи: розробка нових функцій у `feature/*`, виправлення помилок у `bugfix/*`, термінові патчі в `hotfix/*` — без порушення стабільної гілки `main`.

**3. Яка команда створює нову гілку?**  
`git branch <назва_гілки>` — створює нову гілку. `git switch -c <назва_гілки>` — створює і одразу перемикається на неї.

**4. Чим відрізняються команди git branch і git switch?**  
`git branch` створює або показує гілки. `git switch` перемикається на іншу гілку. `git switch -c` комбінує обидві дії.

**5. Що таке merge?**  
Merge (злиття) — це операція об'єднання змін з однієї гілки в іншу (наприклад, `git merge feature/login-form` додає зміни з гілки `feature/login-form` у поточну гілку).

**6. У яких випадках виникає конфлікт злиття?**  
Коли в різних гілках було змінено один і той самий рядок одного й того ж файлу. Git не може автоматично визначити, який варіант залишити.

**7. Як Git позначає конфлікт у файлі?**  
Маркерами:
```
<<<<<<< HEAD
... ваші зміни ...
=======
... зміни з іншої гілки ...
>>>>>>> назва-гілки
```

**8. Як завершити розв'язання конфлікту?**  
1. Відредагувати файл, прибравши маркери. 2. Виконати `git add <файл>`. 3. Виконати `git commit -m "..."`.

**9. Для чого використовується команда git log --graph --all?**  
Показує графічне дерево історії комітів з усіма гілками, що допомагає візуально побачити структуру злиттів та розгалужень.

**10. Які переваги має командна робота через окремі гілки?**  
Паралельна розробка без блокування, ізоляція експериментальних змін, контроль якості через pull request та код-рев'ю, можливість відкотити окрему функцію без впливу на весь проєкт.
